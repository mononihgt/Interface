# 数据结构与处理边界

Interface V2 将稳定的参与者身份、一次参与尝试、每日实验、对话轮次和评分拆分保存。这一结构避免后续重新分配或补做实验覆盖历史记录。

## 核心记录

| 记录 | 用途 | 处理要点 |
| --- | --- | --- |
| `participants` | 稳定的参与者身份和当前参与尝试引用 | 包含可识别信息，属于受保护数据 |
| `participant_attempts` | 每一次参与的分配、状态和导出资格 | 是当前实验流程的事实来源 |
| `participant_days` | 多日实验中的具体实验日 | 与参与尝试关联，保留完成状态 |
| `pretest_responses` | 前测问卷的草稿与最终答案 | 仅在满足研究授权时导出 |
| `experiment_sessions` | 一次正式或测试会话 | 保存会话级实验状态与条件快照 |
| `conversation_turns` | 每轮用户输入、AI 回复、时序与错误证据 | 同一行关联输入、回复和模型路由结果 |
| `turn_ratings` | 对话轮次评分 | 通过 `turn_id` 关联对话轮次 |
| `task_artifacts` | 可展示的执行任务结果 | 与会话和对话轮次关联 |
| `admin_events` | 管理操作审计 | 不应保存密码、密钥、Cookie 或会话令牌 |

## 会话与轮次

`experiment_sessions` 冻结会话创建时的实验条件，并区分测试与正式数据。`conversation_turns` 使用 `turn_index` 表示轮次，将用户文本、展示给参与者的回复、所用模型路由、错误呈现与生成/评估证据集中保存。

主要时序字段如下：

| 字段 | 含义 |
| --- | --- |
| `client_message_sent_at` | 浏览器发送消息前记录的绝对时间 |
| `assistant_render_completed_at` | 浏览器完成回复与结果绘制的绝对时间 |
| `client_response_latency_ms` | 使用浏览器单调时钟测量的被试等待时长 |
| `client_timing_interrupted` | 计时期间页面是否进入后台 |
| `response_latency_ms` | 后端最终成功模型调用的耗时，不等同于被试等待时长 |

分析等待时间时，应以 `client_response_latency_ms` 为主，并排除或单独处理 `client_timing_interrupted=true` 的记录。绝对客户端时间可能受设备时钟影响，不应替代单调时钟时长。

## 计划错误与任务产物

计划错误的计划、参与者可见结果、评估和生成尝试以独立字段保存。`error_planned` 表示计划轮次，`error_presented` 表示是否实际呈现，`error_presentation` 描述呈现方式。未成功呈现的计划错误不应被当作已实现的实验操纵。

`task_artifacts` 是独立的任务结果记录，而不是嵌入在 AI 文本中的未结构化字段。导出或分析时，应通过会话、轮次和产物关系连接数据，避免以展示文案反推状态。

## 导出与安全

导出前应根据实验协议筛选正式、有效且允许导出的参与尝试。测试会话、无效尝试、运行日志和内部证据应按研究目的分别处理，不能默认混入分析数据。

`data/` 可包含身份信息、问卷、对话、音频、转写、导出和日志，始终属于敏感运行时数据：

- 仅授予完成工作所需的最小访问权限。
- 传输、备份和长期保存均遵循适用的数据治理与伦理要求。
- 不将真实数据复制进代码仓库、问题单、截图、测试夹具或公开文档。
- 共享分析数据前，执行必要的去标识化、范围审查和审批流程。
